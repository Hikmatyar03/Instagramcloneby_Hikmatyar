This document details the MongoDB database schema design, entity relationships, and core data flow patterns of the Instagram clone application.

3. Key Design Talking Points (For Presentation)

Why NoSQL / MongoDB?

Instagram's data is highly document-oriented. A post with multiple images (carousels), stickers, and metadata fits perfectly into a single JSON/BSON document. We can embed smaller arrays (like media items or user notification settings) directly inside the document. This removes the need for complex, slow SQL JOIN operations for every single photo attached to a post.

How did you handle indexing?

We added compound indexes on frequently queried fields to ensure the app stays fast even with millions of records. For example, queries for the feed search by user_id and sort by created_at. Therefore, an index on { user_id: 1, created_at: -1 } is applied to the posts collection to instantly return the newest posts from specific users.

How do you handle Likes without overloading the posts document?

We used a Polymorphic Pattern. Instead of putting a massive array of liked_by users inside the Post document itself (which could exceed MongoDB's 16MB document size limit for viral posts), we created a separate likes collection. This collection references the target_id and target_type (e.g., post or comment). A simple counter (likes_count) is kept on the post document for quick UI rendering without needing to count the rows.

Security Approach

Passwords are never stored in plain text. They are hashed using bcrypt (with 12 salt rounds) before ever touching the database. For authentication, JWT (JSON Web Tokens) are used. The Access Token is short-lived for security, and the Refresh Token is stored in an HttpOnly cookie to prevent XSS attacks.
